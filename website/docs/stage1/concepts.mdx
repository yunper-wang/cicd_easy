## 什么是 GitOps？

GitOps 是一种现代化的持续交付方法，其核心思想是 **将 Git 作为基础设施和应用配置的唯一可信源**。它由 Weaveworks 在 2017 年提出，现已成为云原生领域的事实标准。

### GitOps 四大原则

#### 1. 声明式配置（Declarative Configuration）

整个系统的期望状态（Desired State）用声明式语言描述，而非命令式脚本。在 Kubernetes 生态中，这意味着所有资源都用 YAML 清单定义：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    spec:
      containers:
      - name: app
        image: registry.local/my-app:v1.0.0
```

声明式配置的好处：
- **可审计** — 每次变更都有 Git commit 记录
- **可回滚** — `git revert` 即可恢复到之前的版本
- **可复制** — 同一份配置可以在多个环境中复用

#### 2. Git 单一可信源（Single Source of Truth）

系统的期望状态存储在 Git 仓库中，所有变更通过 Git 提交触发。这意味着：
- 任何配置变更都必须经过 Git 流程（commit、review、merge）
- Git 历史记录就是完整的变更审计日志
- 分支保护策略可以强制实施审批流程

#### 3. 自动同步（Automated Sync）

控制器持续监控 Git 仓库，一旦检测到新的提交，自动将变更应用到集群。Argo CD 就是这样一个控制器，它的同步机制包括：
- **周期性轮询** — 默认每 3 分钟检查一次 Git 仓库
- **Webhook 触发** — Git 仓库推送事件可立即触发同步
- **手动同步** — 在 Dashboard 或 CLI 手动触发

#### 4. 持续协调（Continuous Reconciliation）

控制器持续比对 Git 中的期望状态与集群的实际状态，任何偏差都会被自动修正：

| 偏差类型 | 示例 | 协调行为 |
|----------|------|----------|
| 配置漂移 | 有人手动修改了副本数 | 自动恢复到 Git 定义的数量 |
| 资源缺失 | Pod 被意外删除 | 自动重建缺失的资源 |
| 版本不一致 | 镜像标签被手动更改 | 自动恢复到 Git 指定的版本 |

## CI/CD Pipeline 基础

CI/CD Pipeline 是实现 GitOps 的关键引擎。在 GitLab CI 中，Pipeline 由以下核心概念组成：

### Stages（阶段）

Stage 定义了 Pipeline 的执行顺序。同一 Stage 中的 Job 并行执行，不同 Stage 按顺序串行：

```yaml
stages:
  - build    # 第一阶段：构建
  - test     # 第二阶段：测试
  - deploy   # 第三阶段：部署
```

### Jobs（任务）

Job 是 Pipeline 的最小执行单元，每个 Job 必须归属于一个 Stage：

```yaml
build-image:
  stage: build
  script:
    - echo "Building Docker image..."
  only:
    - main

run-unit-tests:
  stage: test
  script:
    - echo "Running tests..."
  only:
    - main
```

### Dependencies（依赖）

通过 `needs` 关键字可以定义 Job 之间的显式依赖关系，创建 DAG（有向无环图）执行模式：

```yaml
build-image:
  stage: build
  script:
    - /kaniko/executor --destination=$CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA

run-unit-tests:
  stage: test
  needs: [build-image]
  script:
    - echo "Testing the built image..."

deploy-to-dev:
  stage: deploy
  needs: [run-unit-tests]
  script:
    - echo "Deploying to dev environment..."
```
